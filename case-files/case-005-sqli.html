// CASE-005 · SOC127 True Positive Cerrado Medium

SQL Injection + XSS + RCE —
Ataque Automatizado a Aplicación Web

Solicitud HTTP maliciosa detectada contra servidor web interno combinando SQL Injection, Cross-Site Scripting y ejecución de comandos en un único payload automatizado. El servidor respondió con HTTP 200 — impacto real indeterminado por falta de acceso a logs del endpoint.

// 01_resumen

Resumen ejecutivo

El IDS detectó una solicitud GET maliciosa originada desde 118.194.247.28 (9/94 detecciones en VirusTotal) dirigida al servidor web interno 172.16.20.12. El payload combina en una sola solicitud técnicas de SQL Injection, Cross-Site Scripting y ejecución de comandos — patrón característico de herramientas automatizadas de escaneo de vulnerabilidades.

El servidor respondió con HTTP 200, lo que indica que procesó la solicitud. Sin acceso a los logs del servidor web ni de la base de datos, no es posible confirmar si alguna de las técnicas fue ejecutada exitosamente. El impacto se clasifica como indeterminado.

// veredicto

True Positive confirmado. Ataque real de aplicación web con payload malicioso verificado. HTTP 200 no confirma ni descarta éxito de la explotación — requiere revisión de logs del servidor y base de datos para determinar impacto real. Se recomienda escalamiento a Tier 2 y bloqueo de IP atacante.

// 03_payload

Análisis del payload

La solicitud HTTP contenía el siguiente payload en el parámetro GET:

// Solicitud HTTP observada

GET /?douj=3034 AND 1=1
UNION ALL SELECT 1,NULL,'<script>alert("XSS")</script>',table_name
FROM information_schema.tables WHERE 2>1--/**/;
EXEC xp_cmdshell('cat ../../../etc/passwd')# HTTP/1.1

Desglose de cada técnica presente en el payload:

SQL INJECTION
AND 1=1 UNION ALL SELECT ... FROM information_schema.tables
Intento de inyección SQL con UNION para enumerar tablas de la base de datos. information_schema.tables es el catálogo interno de SQL Server y MySQL que lista todas las tablas disponibles.
XSS
<script>alert("XSS")</script>
Payload clásico de Cross-Site Scripting para verificar si la aplicación refleja contenido sin sanitización. Si se ejecuta en el navegador de un usuario confirma vulnerabilidad XSS reflejado.
RCE
EXEC xp_cmdshell('cat ../../../etc/passwd')
xp_cmdshell es un procedimiento almacenado de SQL Server que ejecuta comandos del sistema operativo. Combinado con path traversal (../../) intenta leer /etc/passwd — archivo de usuarios en Linux.
Nota técnica: La combinación de xp_cmdshell (SQL Server) con /etc/passwd (Linux) en el mismo payload indica el uso de una herramienta automatizada que prueba múltiples técnicas sin considerar la coherencia del entorno objetivo. Esto es característico de scanners como sqlmap o herramientas similares.

// 05_acciones

Acciones tomadas

01

Verificación de IP atacante: 118.194.247.28 confirmada como maliciosa en VirusTotal (9/94). Sin registros de ejercicio planificado en el sistema de correo.

02

Análisis del payload HTTP: Desglose de cada técnica presente — SQL Injection (UNION SELECT), XSS (script alert), ejecución de comandos (xp_cmdshell) y path traversal (/etc/passwd). Patrón consistente con herramienta automatizada.

03

Evaluación de respuesta HTTP: Servidor respondió con HTTP 200. Se determinó que este código no confirma éxito del ataque — solo indica que la solicitud fue procesada por la aplicación.

04

Revisión de logs de proxy: Se verificaron registros disponibles para identificar actividad adicional desde la misma IP. Sin acceso a logs del servidor web ni base de datos para confirmar impacto.

05

Clasificación y escalamiento: True Positive confirmado. Escalado a Tier 2 para revisión de logs del endpoint y base de datos. Recomendado bloqueo de IP atacante en firewall perimetral.

// 06_recomendaciones

Recomendaciones

  • Revisar logs del servidor web (access.log, error.log) para determinar si el payload fue reflejado o ejecutado por la aplicación
  • Revisar logs de la base de datos para confirmar si las consultas SQL inyectadas fueron procesadas
  • Bloquear 118.194.247.28 en el firewall perimetral para prevenir intentos adicionales
  • Implementar o revisar reglas del WAF para bloquear patrones de SQL Injection y XSS conocidos
  • Verificar que la aplicación web implemente prepared statements y sanitización de entrada para prevenir SQLi
  • Auditar si xp_cmdshell está habilitado en la base de datos — debería estar deshabilitado por defecto
  • Implementar Content Security Policy (CSP) para mitigar riesgo de XSS

// 07_lecciones

Lecciones aprendidas

True Positive no requiere exfiltración de datos. El hecho de que ocurrió un ataque real con payload malicioso es suficiente para clasificarlo como True Positive, independientemente de si el atacante obtuvo datos o acceso exitoso.

HTTP 200 no confirma éxito del ataque. Un código 200 solo indica que el servidor procesó la solicitud. El impacto real solo puede determinarse revisando logs del servidor web y la base de datos.

Payloads incoherentes delatan herramientas automatizadas. Combinar xp_cmdshell (SQL Server) con /etc/passwd (Linux) en la misma solicitud indica scanner automático — un atacante humano adaptaría el payload al sistema objetivo.

Cuando falta visibilidad, el impacto es indeterminado. Sin logs del endpoint no se puede afirmar que el ataque falló. La respuesta correcta es escalar para obtener esa visibilidad, no asumir que todo está bien.
