// 01_objetivo

Objetivo

Construir un entorno de laboratorio SOC funcional con hardware disponible — sin presupuesto para servidores ni instancias cloud — capaz de recolectar logs, detectar eventos de seguridad y correlacionar alertas con el framework MITRE ATT&CK.

La restricción de recursos fue parte del desafío: máquinas con 8 GB y 2 GB de RAM, sin posibilidad de correr múltiples VMs pesadas simultáneamente. La solución tenía que ser eficiente y real.

// 02_entorno

Entorno y arquitectura

Zorin OS Ubuntu Server 24.04 Wazuh 4.7.5 OpenSearch Filebeat Kali Linux VirtualBox SSH

// 03_proceso

Proceso de implementación

01
Instalación de Ubuntu Server en la laptop secundaria

Grabación de ISO en pendrive con Balena Etcher, configuración de red WiFi mediante netplan y primera conexión SSH desde la laptop principal. Primer obstáculo resuelto: adaptador microSD causaba checksum failure durante la instalación.

02
Configuración de red y SSH

Asignación de IPs fijas por MAC en el router doméstico (192.168.0.105 y 192.168.0.103). Instalación de openssh-server en la HP para administración remota sin necesidad de teclado físico.

03
Despliegue de Wazuh con el instalador oficial

Uso del script wazuh-install.sh -a -i para instalación all-in-one de manager, indexer, dashboard y filebeat. Resolución del problema de compatibilidad con Zorin OS usando el flag --ignore-check.

04
Registro del agente en la HP

Instalación del wazuh-agent versión 4.7.5 (compatible con el manager) en la laptop secundaria. Corrección manual de la IP del manager en ossec.conf después de identificar que la variable de entorno no se aplicó correctamente.

05
Verificación y generación de primera alerta

Confirmación de agente activo en el dashboard. Primera alerta real: autenticación SSH desde Kali Linux con Hydra generando 416+ eventos de brute force detectados con mapeo automático a MITRE ATT&CK.

// 04_desafios

Desafíos encontrados y soluciones

Esta sección documenta los problemas reales encontrados durante el despliegue. En un entorno SOC, la capacidad de diagnóstico ante fallos inesperados es tan importante como el conocimiento teórico.

Checksum failure durante instalación de Ubuntu

El instalador fallaba con "install media checksum verification failed" de forma repetida incluso con ISO verificado.

▸ Causa: adaptador microSD a USB inestable. Solución: usar directamente una USB de 8gb para evitar inestabilidad.

Layout de teclado incorrecto en Ubuntu Server

Ubuntu Server se instaló con layout en inglés, imposibilitando escribir caracteres especiales como guiones y barras necesarios para los comandos.

▸ Solución: reinstalación con configuración correcta de localización desde el menú del instalador.

Wazuh Indexer fallando por certificados SSL faltantes

Al instalar componentes por separado, el indexer no encontraba root-ca.pem y fallaba al iniciar. Error: "Unable to read the file /etc/wazuh-indexer/certs/root-ca.pem".

▸ Solución: generar certificados con wazuh-certs-tool.sh y copiarlos manualmente con permisos correctos (400). Lección: usar el instalador oficial all-in-one en lugar de componentes separados.

Saturación de RAM con todos los servicios activos

Wazuh Manager + Indexer + Dashboard consumían ~5.7 GB de los 7.7 GB disponibles, dejando sin margen para herramientas de ataque.

▸ Solución: ampliar swap de 2 GB a 4 GB y gestionar servicios manualmente — parar indexer y dashboard al usar Kali, reactivarlos al analizar alertas.

Alertas generadas pero no visibles en el dashboard

El manager generaba alertas en alerts.json pero el índice wazuh-alerts-* no se creaba en el indexer. Múltiples causas encadenadas: keystore sin credenciales, IP incorrecta del indexer en ossec.conf, Filebeat crasheando.

▸ Solución final: reinstalación completa con wazuh-install.sh -a -i -o que configura correctamente todos los componentes y sus dependencias.

// 05_resultados

Resultados — Primera alerta real

Al ejecutar un ataque de fuerza bruta SSH con Hydra desde Kali Linux contra la laptop HP, el dashboard mostró 416+ alertas de autenticación. Ejemplo de un evento capturado:

@timestamp 2026-05-26T19:45:10.707Z agent.name ubuntuserver agent.ip 192.168.0.103 rule.id 5501 rule.description PAM: Login session opened. rule.level 3 rule.mitre.id T1078 rule.mitre.tactic Defense Evasion · Persistence · Privilege Escalation · Initial Access rule.mitre.techniqueValid Accounts rule.pci_dss 10.2.5 rule.gdpr IV_32.2 rule.nist_800_53 AU.14 · AC.7 location /var/log/auth.log

El SIEM mapea automáticamente cada evento a MITRE ATT&CK, frameworks de cumplimiento (PCI DSS, GDPR, HIPAA, NIST 800-53) y asigna nivel de severidad. Exactamente la visibilidad que tiene un analista SOC Tier 1 en su trabajo diario.

// 06_lecciones

Lecciones aprendidas

Los instaladores oficiales existen por algo. Intentar instalar componentes de Wazuh por separado genera problemas de certificados, versiones y dependencias que el script oficial resuelve automáticamente.

Los errores son el laboratorio real. Cada fallo de checksum, permiso denegado o servicio caído fue una oportunidad de diagnóstico que ninguna guía perfecta hubiera enseñado.

La gestión de recursos es una habilidad operativa. Saber qué servicios pausar y cuándo, cómo ampliar swap, cómo priorizar procesos — todo esto es administración de sistemas real.

SSH es la herramienta más importante del lab. Poder administrar la laptop secundaria de forma remota sin necesitar teclado ni pantalla transformó completamente la experiencia de trabajo.

Wazuh correlaciona más de lo esperado. Sin configuración adicional, el agente detectó automáticamente eventos PAM, sudo, gestión de usuarios y los mapeó a MITRE ATT&CK con cumplimiento normativo incluido.

El hardware limitado no es excusa. Con dos laptops viejas, swap ampliada y gestión manual de servicios se construyó un entorno SOC completamente funcional.
